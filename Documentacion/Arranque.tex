\section{Lista de características}

Siguiendo la fase 1 de FDD (Desarrollar lista de características) y los objetivos definidos para el proyecto, se han identificado las siguientes funcionalidades nucleares priorizadas por valor para el usuario:

\begin{enumerate}
    \item \textbf{Gestión de Usuarios (OAuth2)}: Autenticación delegada mediante Google y persistencia de perfil (nombre, foto). El usuario puede personalizar su nombre y foto de perfil desde la página de perfil, con soporte para eliminar la cuenta.
    \item \textbf{Mapa Interactivo}: Visualización de un mapa navegable mediante MapLibre GL JS como superficie central de interacción, con búsqueda de lugares integrada (Nominatim).
    \item \textbf{Escáner de Especies}: Identificación de fauna marina en una zona mediante radio de búsqueda y validación cruzada de APIs científicas (OBIS + iNaturalist), con sistema de caché adaptativo (TTL 48h) para reducir latencia y llamadas externas.
    \item \textbf{Datos Climáticos y Oceanográficos}: Condiciones de la zona en tiempo real mediante Open-Meteo (viento, temperatura del agua, corrientes, olas, datos atmosféricos), con evaluación automática de aptitud para el buceo y caché de 30 minutos.
    \item \textbf{Registro de Publicaciones}: Publicación de inmersiones en el mapa con título, descripción, fotografía y geolocalización inversa (nombre del lugar mediante Nominatim). Edición y borrado por el propio autor.
    \item \textbf{Interacciones Sociales}: Sistema de likes, guardados y comentarios sobre publicaciones. Soporte de menciones en comentarios mediante la sintaxis \texttt{@[Nombre](email)} con notificaciones automáticas al mencionado.
    \item \textbf{Red Social y Seguimiento de Usuarios}: Perfiles públicos por usuario con recuento de seguidores y seguidos, posibilidad de seguir/dejar de seguir, y vista de publicaciones de cada perfil. Página de perfil propio con mis publicaciones y publicaciones guardadas.
    \item \textbf{Sistema de Notificaciones}: Notificaciones en tiempo real (campana en el header) para eventos de seguimiento, like, comentario y mención. Las notificaciones se marcan como leídas al abrirlas o se eliminan permanentemente con swipe (móvil) o botón (escritorio).
\end{enumerate}

\textbf{Justificación de priorización}:
\begin{itemize}
    \item La autenticación se implementa primero ya que el resto de funcionalidades (avistamientos, perfil, red social) dependen de la identidad del usuario.
    \item El mapa es la superficie central de la aplicación: sin él no tiene sentido geolocalizar especies, condiciones climáticas ni avistamientos.
    \item El escáner de especies y los datos climáticos constituyen el MVP funcional mínimo: son las dos propuestas de valor diferenciales respecto a las aplicaciones analizadas en el capítulo de Contexto.
    \item El registro de avistamientos, la red social y el perfil compartible son las funcionalidades de comunidad que convierten la herramienta en una red social de buceadores.
\end{itemize}

\section{Diseño arquitectónico}
\label{sec:arquitectura}

Con las características identificadas, el sistema requiere una arquitectura capaz de orquestar múltiples APIs externas, gestionar identidad de usuario y servir una interfaz cartográfica interactiva. Se ha optado por una \textbf{arquitectura cliente-servidor desacoplada} con patrón Gateway de APIs.

\subsection{Stack Tecnológico}

\begin{itemize}
    \item \textbf{Backend}: Spring Boot 4 (Java 21). Actúa como Gateway de APIs científicas, orquestador de peticiones asíncronas y gestor de autenticación.
    \item \textbf{Frontend}: React 18 + Vite. Con Hot Module Replacement (HMR) para desarrollo ágil.
    \item \textbf{Cartografía}: MapLibre GL JS. Motor de renderizado vectorial para animaciones fluidas y bajo consumo de datos.
    \item \textbf{Base de Datos}: H2 en memoria para el entorno de desarrollo y tests unitarios. PostgreSQL 16 para preproducción y producción, gestionado como servicio en Railway.
    \item \textbf{Seguridad}: Spring Security con OAuth2 Client para Google Login.
    \item \textbf{Documentación de API}: OpenAPI 3.0 (Swagger UI) con anotaciones automáticas.
    \item \textbf{Despliegue}: Railway (backend + PostgreSQL) y Vercel (frontend).
\end{itemize}

\textbf{Justificación de la separación entre plataformas}:
\begin{itemize}
    \item \textbf{Vercel} gestiona el frontend React/Vite: Se despliega automáticamente en cada \texttt{push}. Se cuenta con previews por rama sin configuración adicional, más que establecer las direcciones de backend de producción y preproducción.
    \item \textbf{Railway} gestiona el backend Spring Boot y PostgreSQL en el mismo proyecto. Permite tener varios entornos distintos de una misma aplicación desplegados simultaneamente, lo que la hace idonéa para combinarla con Vercel.
    \item La separación permite desplegar frontend y backend de forma independiente. Gracias a esto pueden escalarse independientemente y no solo eso. Debido a la separación se cuenta con mas memoria de los planes gratuitos de ambas aplicaciones, reduciendo costes. 
\end{itemize}

La configuración CORS se ha establecido manualmente mediante la variable \texttt{CORS\_ALLOWED\_ORIGIN} en Railway, y \texttt{VITE\_API\_BASE\_URL} en Vercel, configuradas con las respectivas URL del backend correspondientes a preproducción y producción.

\subsection{Justificación de MapLibre GL JS}

MapLibre GL JS se ha seleccionado porque:
\begin{itemize}
    \item \textbf{Sin necesidad de geocodificación}: El proyecto no requiere búsqueda de direcciones ni rutas, solo visualización de capas geoespaciales.
    \item \textbf{Open Source y sin rate limits}: Alternativa libre a Mapbox GL (del cual es fork).
\end{itemize}

\subsection{Patrón Arquitectónico: Gateway API}

El backend actua como director de orquesta de las API, gestionando la conexión entre las mismas. Pasándole al frontend unos datos ya procesados en formato JSON, que el frontend guarda en una store. Este es el flujo de datos desde el principio:

\begin{enumerate}
    \item \textbf{El usuario clicka una zona}. Y react realiza esta llamada \texttt{GET /api/species?lat=X\&lng=Y\&radius=Z}.
    \item \textbf{SpeciesService} (backend) realiza:
    \begin{enumerate}
        \item Una transformación geométrica, convirtiendo el radio a un polígono: \texttt{(lat, lng, radius)} $\rightarrow$ \texttt{WKT POLYGON}. Esto se realiza porque en la...
        \item Petición a OBIS, hace falta pasar un polígono de la zona. De la cual se extraeran los datos de las especies. La llamada a OBIS es la siguiente: \texttt{GET /occurrence?geometry=POLYGON(...)}.
        \item Para cada especie única devuelta por OBIS se realiza un procesamiento de datos:
        \begin{itemize}
            \item Primero se hace una petición a iNaturalist: \texttt{GET /taxa?q=<scientificName>}. Esta llamada devuelve datos interesantes de la  especie que serviran mas adelante.
            \item Segundo: Filtrado, si \texttt{total\_results == 0}, se descarta la especie por completo. La mayoría de especies que no se encuentran en iNaturalist son microorganismos sin interés para buceadores recreativos. 
            \item Tercero: Aquí se extraen los datos interesantes. Una foto y nombre común. Para que todas las personas puedan saber que especie se ha registrado 
        \end{itemize}
    \end{enumerate}
    \item \textbf{Finalmente se unifica la respuesta y se crea un JSON para el frontend}. Se hace mediante el DTO \texttt{SpeciesResponse}.
\end{enumerate}

\textbf{Ventajas}:
\begin{itemize}
    \item React no necesita conocer la existencia de múltiples APIs.
    \item La lógica de filtrado (microorganismos, especies sin foto) se centraliza en el backend.
    \item Se evita exposición de API keys en el navegador.
\end{itemize}
