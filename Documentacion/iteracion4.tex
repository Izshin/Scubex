\chapter{Iteración 4: Interacciones y Red Social}

\begin{abstract}
Esta iteración convierte Scubex en una red social de buceo. Se añaden likes, guardados y comentarios con menciones en las publicaciones, la posibilidad de seguir a otros buceadores con perfiles públicos, y un centro de notificaciones que informa al usuario de toda actividad relacionada con su contenido.\end{abstract}

% ─────────────────────────────────────────────────────────────────────────────
\section{Características desarrolladas esta iteración}
% ─────────────────────────────────────────────────────────────────────────────

\begin{enumerate}
\item Interacciones con publicaciones: likes, guardados y comentarios con soporte para menciones a otros usuarios. Ver Tabla~\ref{tab:valorAportadoInteracciones}.
\item Seguimiento de usuarios y perfiles públicos: seguir a otros buceadores y consultar su perfil con contadores de seguidores y seguidos. Ver Tabla~\ref{tab:valorAportadoFollows}.
\item Centro de notificaciones: campana con contador de no leídas que agrupa likes, comentarios, menciones y nuevos seguidores. Ver Tabla~\ref{tab:valorAportadoNotificaciones}.
\end{enumerate}
Con las publicaciones en el mapa, el siguiente paso natural es que los usuarios puedan reaccionar al contenido de los demás. Las interacciones dan retroalimentación al autor; los follows construyen la red entre buceadores; y las notificaciones cierran el ciclo informando al usuario de todo lo que ocurre con su contenido, sin que tenga que ir a buscarlo.

\section{Implementación}
% ─────────────────────────────────────────────────────────────────────────────

La capa social introduce tres desafíos que no existían en iteraciones anteriores: gestionar acciones reversibles sin duplicar datos, enviar notificaciones precisas sin repetirlas, y mantener el contador actualizado sin conexiones permanentes al servidor. Los flujos siguientes describen cómo se resuelven en la práctica; las tablas al final de la sección recogen las decisiones de diseño detrás de cada uno.

\subsection*{Flujo de una interacción (like)}

Cuando un usuario pulsa el botón de like en una publicación:

\begin{enumerate}
    \item El usuario pulsa el botón. La interfaz actualiza el contador y el estado visual del botón de forma inmediata sin esperar respuesta del servidor (actualización optimista). Al mismo tiempo, se envía la petición y el servidor comprueba si ya existe un like de ese usuario para esa publicación en base de datos.
    \item Si ya existe, lo borra y devuelve al cliente el nuevo estado (sin like). No se genera ninguna notificación.
    \item Si no existe, crea el registro y notifica al autor de la publicación. Si el usuario está dando like a su propia publicación, o ya había recibido una notificación de like del mismo actor, no se genera ninguna notificación adicional.
    \item Si la respuesta del servidor no coincide con el estado optimista, la interfaz lo corrige para reflejar la realidad.
\end{enumerate}

\smallskip
\noindent\textit{Nota: el flujo de guardados y el de follows siguen exactamente el mismo patrón de toggle. Las diferencias son dos: los guardados no generan ninguna notificación (es una acción privada del usuario), y el follow añade una comprobación extra que impide que un usuario se siga a sí mismo antes de llegar a la base de datos.}

\subsection*{Flujo de un comentario con mención}

Cuando un usuario publica un comentario que menciona a otro usuario:

\begin{enumerate}
    \item El usuario envía el formulario. El frontend añade de inmediato un comentario temporal a la lista, incrementa el contador de comentarios y actualiza el badge del marcador en el mapa, todo sin esperar respuesta del servidor (actualización optimista). La caja de texto se vacía para que el usuario pueda escribir otro comentario.
    \item La petición llega al servidor y el comentario se persiste en base de datos con el texto tal como el usuario lo escribió. Las menciones viajan codificadas como \texttt{@[Nombre](email)}.
    \item El servidor extrae los emails mencionados del texto y los resuelve a usuarios en una sola consulta a base de datos.
    \item Si el comentarista no es el propietario de la publicación, se genera una notificación de tipo comentario para el propietario y se lo marca como ya notificado.
    \item Por cada usuario mencionado que no haya sido ya notificado en el paso anterior, se genera una notificación de tipo mención. Esto evita que el propietario reciba dos notificaciones si también aparece mencionado.
    \item El servidor devuelve el comentario real con el identificador definitivo. El frontend sustituye el comentario temporal por el definitivo. Si la petición falló, deshace todos los cambios optimistas: elimina el comentario temporal, restaura el contador y devuelve el texto al campo de entrada.
\end{enumerate}

\smallskip
\noindent\textit{Nota: un comentario sin mención sigue exactamente el mismo camino. La única diferencia es que la lista de emails mencionados queda vacía, por lo que no se generan notificaciones de mención.}

\textbf{Asunto}&El usuario debe ver las notificaciones nuevas sin tener que recargar la página.\\
\textbf{Resumen}&La campana consulta periódicamente al servidor cuántas notificaciones sin leer hay y actualiza el contador. Al abrirla, descarga la lista completa.\\

\textbf{Factores causantes}&Las notificaciones las generan acciones de otros usuarios, por lo que el cliente no sabe cuándo llega una nueva. Sin consultas periódicas el contador solo cambiaría al recargar.\\

\textbf{Motivación}&El polling no requiere infraestructura adicional y es suficiente para el volumen de usuarios esperado. La pequeña latencia que introduce es irrelevante en la práctica.\\
\textbf{Cuestiones abiertas}&El intervalo de polling fija el retraso máximo con el que el usuario ve una notificación nueva. No hay alertas del sistema operativo cuando la pestaña está en segundo plano.\\
\textbf{Alternativas}&WebSockets para tiempo real completo, pero añaden complejidad de infraestructura innecesaria para el volumen de Scubex.\\

\section{Despliegue}
% ─────────────────────────────────────────────────────────────────────────────

Esta iteración añadió cinco nuevas tablas a la base de datos: \texttt{publication\_likes}, \texttt{publication\_saves}, \texttt{comments}, \texttt{user\_follows} y \texttt{notifications}, todas creadas automáticamente por Hibernate al arrancar el servicio en Railway. No fue necesario añadir nuevas variables de entorno ni credenciales externas, ya que todas las funcionalidades de esta iteración utilizan exclusivamente la base de datos PostgreSQL existente.

\section{Pruebas}
% ─────────────────────────────────────────────────────────────────────────────

Las pruebas cubren los tres desafíos técnicos documentados y verifican los pasos clave de los dos flujos.

\subsection*{Interacciones}

\begin{itemize}
    \item \textbf{Primer like} (Tabla~\ref{tab:memo012}, paso~3 del flujo de like): sin like previo, se persiste el registro y se notifica al autor.

    \item \textbf{Quitar like} (Tabla~\ref{tab:memo012}, paso~2 del flujo de like): si el like ya existe, se elimina y no se genera ninguna notificación.

    \item \textbf{Primer guardado y quitar guardado} (Tabla~\ref{tab:memo012}): mismo patrón que los likes, verificando que el registro se crea o se elimina según corresponda.

    \item \textbf{Añadir comentario} (Tabla~\ref{tab:memo012}, paso~2 del flujo de comentario): el comentario se persiste y se delega la notificación.

    \item \textbf{Borrar comentario por el autor} (Tabla~\ref{tab:memo012}): el autor borra su comentario; resultado verdadero y borrado ejecutado.

    \item \textbf{Borrar comentario por usuario ajeno} (Tabla~\ref{tab:memo012}): un usuario distinto al autor obtiene falso y no se ejecuta ningún borrado.

    \item \textbf{Borrar comentario inexistente} (Tabla~\ref{tab:memo012}): identificador no encontrado devuelve falso sin intentar borrar.
\end{itemize}

\subsection*{Notificaciones y menciones}

\begin{itemize}
    \item \textbf{Notificación de follow} (Tabla~\ref{tab:memo013}): seguir a otro usuario persiste una notificación de tipo seguimiento con el email del seguidor.

    \item \textbf{Auto-follow sin notificación} (Tabla~\ref{tab:memo013}): seguirse a uno mismo no genera ninguna notificación.

    \item \textbf{Deduplicación de follow y like} (Tabla~\ref{tab:memo013}): si ya existe una notificación del mismo actor para el mismo destinatario y tipo, no se persiste ninguna adicional.

    \item \textbf{Like propio} (Tabla~\ref{tab:memo013}): dar like a la propia publicación no genera notificación.

    \item \textbf{Comentario con mención} (Tabla~\ref{tab:memo013}, pasos~3--4 del flujo de comentario): un comentario con mención produce una notificación de comentario para el propietario y una de mención para el mencionado.

    \item \textbf{Mención al propietario sin duplicar} (Tabla~\ref{tab:memo013}, paso~4 del flujo de comentario): si el propietario es también el mencionado, solo se genera una notificación, no dos.

    \item \textbf{Texto largo truncado} (Tabla~\ref{tab:memo013}, paso~5 del flujo de comentario): un comentario de más de 150 caracteres produce un fragmento de exactamente 151 (150~+~«\ldots»).

    \item \textbf{Marcar como leída y eliminar: solo el destinatario} (Tabla~\ref{tab:memo014}): marcar como leída o eliminar una notificación ajena no persiste ningún cambio; las mismas operaciones sobre la propia notificación sí funcionan.
\end{itemize}
