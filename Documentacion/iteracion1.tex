\chapter{Iteración 1: Inicio del MVP de Scubex}

\begin{abstract}
Esta iteración entrega parte del MVP funcional de Scubex: el mapa interactivo 
con escáner de biodiversidad marina y autenticación con Google.\end{abstract}

% ─────────────────────────────────────────────────────────────────────────────
\section{Características desarrolladas esta iteración}
% ─────────────────────────────────────────────────────────────────────────────

\begin{enumerate}
\item Mapa interactivo con capas de teselas, batimetría y búsqueda de lugares. Ver Tabla~\ref{tab:valorAportadoMapa}.
\item Escáner de biodiversidad marina con enriquecimiento OBIS\,+\,iNaturalist. Ver Tabla~\ref{tab:valorAportadoScan}.
\item Autenticación con Google OAuth2 y emisión de JWT. Ver Tabla~\ref{tab:valorAportadoAuth}.
\end{enumerate}

Esta iteración construye el núcleo de Scubex. El punto de partida es el mapa: sin una superficie interactiva donde el usuario pueda moverse y fijar zonas, ninguna otra funcionalidad tiene sentido. Sobre ese mapa se monta el escáner de biodiversidad, que es parte de la propuesta de valor diferencial de la aplicación. Y para que el usuario pueda guardar y compartir lo que encuentra, se añade la autenticación. Las tres características se analizan a continuación.

\section{Implementación}
% ───────────────────────────────────────────────────────────────────────────────

Para implementar el escáner se consultaron dos APIs externas (OBIS e iNaturalist) y se diseñó una caché de dos niveles para mantener los tiempos de respuesta aceptables. La autenticación se delegó en Google para evitar gestionar credenciales propias. Los cinco desafíos técnicos y las decisiones adoptadas para resolverlos están recogidos en las tablas siguientes:

\begin{itemize}
    \item \textbf{Geometría de búsqueda} (Tabla~\ref{tab:memo001}): OBIS exige un polígono WKT, no un radio. Se genera uno de 12 vértices a partir del punto y radio elegidos por el usuario.
    \item \textbf{Filtrado de calidad} (Tabla~\ref{tab:memo002}): OBIS devuelve microorganismos sin valor visual. Se descartan las especies sin entrada en iNaturalist.
    \item \textbf{Caché de zona} (Tabla~\ref{tab:memo003}): un escaneo puede tardar 10--20\,s. Se persiste el resultado por zona durante 48\,h para evitar repetir las llamadas.
    \item \textbf{Caché de enriquecimiento} (Tabla~\ref{tab:memo004}): la misma especie en otra zona volvería a llamar a iNaturalist. Se guarda por nombre científico durante 30 días.
    \item \textbf{Autenticación OAuth2 + JWT} (Tabla~\ref{tab:memo005}): sin servidor de sesiones, la identidad viaja en cada petición. Se valida el token de Google y se emite un JWT propio de 7 días.
\end{itemize}

\subsection*{Flujo del escáner de biodiversidad}

Con las cuatro decisiones en su lugar, el flujo completo al pulsar el botón de escaneo es el siguiente:

\begin{enumerate}
    \item El servidor redondea las coordenadas del punto elegido y comprueba si existe un escaneo reciente de esa zona en base de datos. Si lo hay, devuelve el resultado directamente sin llamar a ninguna API.
    \item Si no hay caché, construye el polígono WKT y consulta OBIS para obtener las especies registradas en esa zona.
    \item Para cada especie, comprueba la caché de enriquecimiento: si ya existe, se recupera al instante.
    \item Las especies nuevas se consultan en iNaturalist con un máximo de tres llamadas en paralelo para no saturar el servicio. Las que iNaturalist no reconoce se descartan y se guarda ese descarte durante 30 días para no volver a consultarlas.
    \item Las especies reconocidas, con foto y nombre común, reciben además los datos ecológicos de OBIS (profundidad, temperatura, categoría de conservación) y se guardan en la caché de enriquecimiento.
    \item El resultado completo del escaneo se persiste en base de datos para que la próxima consulta sobre esa zona sea inmediata.
\end{enumerate}

\subsection*{Flujo de autenticación}

El flujo de autenticación con Google es el siguiente:

\begin{enumerate}
    \item El usuario hace clic en ``Continuar con Google'' y autoriza el acceso. Google devuelve un token de identidad al frontend.
    \item El frontend envía ese token al backend. El servidor lo valida directamente contra la API de Google para confirmar que es auténtico y fue emitido para esta aplicación concretamente, no para otra.
    \item Si la validación falla, se rechaza la petición con error de autenticación.
    \item Si es correcta, el servidor busca al usuario en base de datos por su identificador de Google: si ya existe lo recupera; si es la primera vez, lo registra automáticamente.
    \item Se emite un token propio firmado con validez de 7 días. El cliente lo guarda y lo incluye en todas sus peticiones posteriores como credencial.
\end{enumerate}

\section{Despliegue}
% ─────────────────────────────────────────────────────────────────────────────

Al cierre de esta iteración la aplicación quedó desplegada en la infraestructura descrita en el Arranque del proyecto:

\begin{itemize}
    \item \textbf{Backend (Railway)}: el proceso de arranque ejecuta \texttt{mvn package} y lanza el JAR resultante. Las variables de entorno \texttt{GOOGLE\_CLIENT\_ID}, \texttt{JWT\_SECRET}, \texttt{DATABASE\_URL} y \texttt{CORS\_ALLOWED\_ORIGINS} se configuran desde el panel de Railway.
    \item \textbf{Frontend (Vercel)}: cada \texttt{push} a \texttt{main} desencadena un despliegue automático. La variable \texttt{VITE\_API\_BASE\_URL} apunta a la URL pública del backend en Railway y \texttt{VITE\_MAPTILER\_KEY} provee acceso a las teselas marineras de MapTiler.
    \item \textbf{Base de datos (PostgreSQL en Railway)}: accedida mediante \texttt{DATABASE\_URL}; Hibernate gestiona el esquema automáticamente (\texttt{ddl-auto=update}).
\end{itemize}

La aplicación es accesible públicamente en \texttt{https://scubex.vercel.app} desde la finalización de esta iteración.

\section{Pruebas}

Tras realizar el código, se han creado tests unitarios que cubren los cinco problemas descritos en la sección de implementación y verifican los pasos clave de los flujos de escáner y autenticación. 

Se han implementado más pruebas unitarias (validación de parámetros de entrada, casos

\subsection*{Escáner de biodiversidad}

\begin{itemize}
    \item \textbf{Geometría WKT} (Tabla~\ref{tab:memo001}, paso~2 del flujo): dado el punto \texttt{(36.5,\,-4.0)} y radio 5000\,m, verifica que la URI generada contiene un \texttt{POLYGON} cerrado con al menos 4 vértices.

    \item \textbf{Filtrado de microorganismos} (Tabla~\ref{tab:memo002}, paso~4 del flujo): OBIS devuelve \textit{Pycnococcaceae}; iNaturalist responde \texttt{total\_results=0}. Verifica que el resultado es una lista vacía.

    \item \textbf{Enriquecimiento con foto} (Tabla~\ref{tab:memo002}, paso~4 del flujo): OBIS devuelve \textit{Chimaera monstrosa}; iNaturalist devuelve nombre común y foto. Verifica que \texttt{commonName} y \texttt{photoUrl} están presentes en la respuesta.

    \item \textbf{Datos ecológicos OBIS} (Tabla~\ref{tab:memo004}, paso~5 del flujo): \textit{Octopus vulgaris} con entrada válida en iNaturalist. Verifica que profundidad, temperatura y categoría IUCN llegan correctamente en el \texttt{SpeciesResponse}.

    \item \textbf{Resiliencia ante fallo parcial de iNaturalist} (paso~4 del flujo): OBIS devuelve \textit{Octopus vulgaris} y \textit{Sepia officinalis}; la primera llamada a iNaturalist lanza \texttt{RestClientException} y la segunda funciona. Verifica que el resultado contiene exactamente una especie con todos sus campos.
\end{itemize}

\subsection*{Autenticación OAuth2 + JWT}

\begin{itemize}
    \item \textbf{Token sin credencial} (Tabla~\ref{tab:memo005}, paso~3 del flujo): petición sin campo \texttt{credential} devuelve \texttt{400 Bad Request}.

    \item \textbf{Token rechazado por Google} (Tabla~\ref{tab:memo005}, paso~3 del flujo): Google no reconoce el token. Verifica que el servidor devuelve \texttt{401 Unauthorized}.

    \item \textbf{Login válido completo} (Tabla~\ref{tab:memo005}, pasos~4--5 del flujo): token válido con \texttt{aud} correcto. Verifica que la respuesta es \texttt{200 OK} con JWT y datos del usuario.

    \item \textbf{JWT: claims correctos y validación} (\texttt{JwtServiceTest}): verifica que el token generado contiene \texttt{googleId}, \texttt{email}, \texttt{name} y \texttt{picture}, y que un token manipulado lanza excepción.
\end{itemize}
