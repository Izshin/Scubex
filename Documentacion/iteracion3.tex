\chapter{Iteración 3: Publicaciones}

\begin{abstract}
Esta iteración añade el final del MVP de Scubex: Un sistema de publicaciones. Los usuarios autenticados pueden crear publicaciones geolocalizadas con imagen desde cualquier punto del mapa, y cualquier visitante puede consultarlas directamente desde los marcadores del mapa.\end{abstract}

% ─────────────────────────────────────────────────────────────────────────────
\section{Características desarrolladas esta iteración}
% ─────────────────────────────────────────────────────────────────────────────

\begin{enumerate}
\item Publicaciones geolocalizadas: crear, editar y eliminar entradas con título, descripción e imagen, ancladas a un punto del mapa. Ver Tabla~\ref{tab:valorAportadoPub}.
\end{enumerate}

Con el mapa operativo y la autenticación en su lugar, la última pieza del MVP es el contenido generado por los propios usuarios. Sin publicaciones, Scubex es una herramienta de consulta; con ellas, se convierte en un diario comunitario de inmersiones donde los usuarios pueden ver que han encontrado otros buceadores en un punto específico del mapa.

\section{Implementación}
% ───────────────────────────────────────────────────────────────────────────────

El sistema de publicaciones necesita escalar con el volumen de contenido sin degradar el rendimiento, ofrecer contexto geográfico legible al usuario y almacenar imágenes de forma persistente sin servicios adicionales. Los tres desafíos técnicos y las decisiones adoptadas para resolverlos están recogidos en las tablas siguientes:

\begin{itemize}
    \item \textbf{Filtrado por área visible} (Tabla~\ref{tab:memo009}): descargar todas las publicaciones en cada consulta del mapa haría la aplicación más lenta conforme crece el contenido. El servidor devuelve solo las publicaciones dentro del área visible, apoyado en un índice de coordenadas.
    \item \textbf{Geocoding inverso} (Tabla~\ref{tab:memo010}): las coordenadas crudas no aportan contexto al usuario. El frontend traduce cada punto a un nombre de lugar mediante Nominatim y guarda el resultado en memoria de sesión para no repetir la consulta.
    \item \textbf{Almacenamiento de imágenes como BLOB} (Tabla~\ref{tab:memo011}): los contenedores de Railway no tienen disco persistente. Las imágenes se guardan directamente en PostgreSQL y se sirven con caché de un año en el navegador.
\end{itemize}

\subsection*{Flujo de creación de una publicación}

Cuando un usuario autenticado hace clic en el mapa y envía el formulario de nueva publicación, el flujo es el siguiente:

\begin{enumerate}
    \item El usuario selecciona el punto en el mapa. El frontend resuelve automáticamente las coordenadas a un nombre de lugar legible mediante Nominatim y lo muestra en el formulario. Si el nombre ya fue consultado en esta sesión, se recupera de la memoria local sin llamar a Nominatim.
    \item Si la publicación incluye una imagen, el frontend la envía primero al servidor. El servidor comprueba que el formato sea aceptado (JPEG, PNG, GIF o WebP) y que no supere los 5\,MB, guarda los bytes en PostgreSQL y devuelve la URL que identifica la imagen.
    \item El usuario confirma el formulario con título, descripción y, si se subió, la URL de la imagen. El servidor verifica que la sesión sigue activa y que los campos obligatorios están presentes.
    \item La publicación se persiste con las coordenadas del punto, el autor y la fecha de creación.
    \item La publicación aparece inmediatamente como marcador en el mapa para cualquier usuario que explore esa zona.
\end{enumerate}

\subsection*{Flujo de consulta de publicaciones}

Cuando el mapa se mueve o se carga por primera vez, las publicaciones visibles se obtienen así:

\begin{enumerate}
    \item El frontend envía al servidor los límites del área visible del mapa (latitud y longitud mínima y máxima).
    \item El servidor consulta la base de datos filtrando por esas coordenadas usando el índice. Solo se devuelven las publicaciones que caen dentro del rectángulo visible.
    \item Cada publicación se devuelve con los datos del autor, las coordenadas y la URL de imagen si la tiene.
    \item El frontend coloca un marcador en el mapa por cada publicación. Al hacer clic en uno, se carga el detalle completo de la publicación.
    \item Si la publicación tiene imagen, el navegador la solicita por su URL. El servidor la sirve con una cabecera de caché de un año: después de la primera descarga, el navegador no vuelve a pedirla.
\end{enumerate}

\textbf{Cuestiones abiertas}&Nominatim limita las peticiones a una por segundo. En sesiones de exploración rápida las peticiones abandonadas se cancelan, pero sin una cola controlada podría alcanzarse el límite.\\
\textbf{Alternativas}&\textbf{Google Maps Geocoding API}: mayor precisión, pero de pago.\\

\section{Despliegue}
% ─────────────────────────────────────────────────────────────────────────────

Esta iteración añadió dos nuevas tablas a la base de datos: \texttt{publications} y \texttt{uploaded\_images}, ambas creadas automáticamente por Hibernate al arrancar el servicio en Railway. No fue necesario añadir nuevas variables de entorno ni credenciales, ya que el almacenamiento de imágenes se realiza en la misma base de datos PostgreSQL existente.

\section{Pruebas}
% ─────────────────────────────────────────────────────────────────────────────

Las pruebas cubren los tres desafíos técnicos documentados y verifican los pasos clave de los dos flujos.

\subsection*{Publicaciones}

\begin{itemize}
    \item \textbf{Filtrado por área} (Tabla~\ref{tab:memo009}, paso~2 del flujo de consulta): el repositorio recibe un bounding box concreto y devuelve dos publicaciones. Verifica que el servicio retorna exactamente esas dos y que la consulta se ejecuta con los límites correctos.

    \item \textbf{Edición por propietario ajeno} (Tabla~\ref{tab:memo009}): un usuario distinto al autor intenta editar la publicación. Verifica que el resultado es nulo y que no se persiste ningún cambio.

    \item \textbf{Eliminación por propietario ajeno} (Tabla~\ref{tab:memo009}): un usuario distinto al autor intenta borrar la publicación. Verifica que el resultado es falso y que no se ejecuta el borrado.
\end{itemize}

\subsection*{Imágenes}

\begin{itemize}
    \item \textbf{Subida sin autenticación} (Tabla~\ref{tab:memo011}, paso~2 del flujo de creación): sin sesión activa, devuelve \texttt{401} sin persistir nada.

    \item \textbf{Validaciones de archivo} (Tabla~\ref{tab:memo011}, paso~2 del flujo de creación): cuatro casos independientes verifican que un archivo vacío, uno superior a 5\,MB, uno sin tipo declarado y uno con formato no permitido (PDF) devuelven \texttt{400} sin persistir nada.

    \item \textbf{Subida válida} (Tabla~\ref{tab:memo011}, paso~2 del flujo de creación): un JPEG válido con usuario autenticado devuelve \texttt{200} con la URL de la imagen y persiste exactamente una imagen en base de datos.

    \item \textbf{Descarga de imagen existente} (Tabla~\ref{tab:memo011}, paso~5 del flujo de consulta): una imagen existente devuelve \texttt{200}, el \texttt{Content-Type} correcto y los bytes originales.

    \item \textbf{Descarga de imagen inexistente} (Tabla~\ref{tab:memo011}, paso~5 del flujo de consulta): un identificador que no existe en base de datos devuelve \texttt{404}.
\end{itemize}
